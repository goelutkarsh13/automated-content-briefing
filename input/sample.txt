Retrieval-augmented generation, often shortened to RAG, is a system design pattern used to improve the factual quality and contextual relevance of language model outputs. Instead of asking a model to rely entirely on what it learned during training, a RAG system first retrieves relevant information from an external knowledge source, such as documents, internal notes, or databases, and then uses that information to guide the generation step.

A typical RAG pipeline includes ingestion, chunking, embedding generation, vector indexing, retrieval, prompt construction, and response synthesis. During ingestion, source documents are cleaned and split into chunks that are small enough to be embedded and searched efficiently. Each chunk is converted into a numerical vector representation using an embedding model, and those vectors are stored in a vector database or similarity index.

When a user asks a question, the system converts the query into an embedding, searches for nearby document chunks, and selects the most relevant evidence. Those retrieved passages are then added to the prompt given to the language model. This allows the model to ground its answer in the retrieved context rather than relying only on internal memory.

RAG systems are valuable because they can reduce hallucinations, improve transparency, and make it easier to update knowledge without retraining the base model. If the underlying knowledge source changes, developers can refresh the index rather than retraining a large model from scratch. This makes the approach especially useful for enterprise assistants, domain-specific question answering, compliance workflows, and research tools.

However, RAG is not a silver bullet. Poor chunking, weak embeddings, irrelevant retrieval, and prompt overload can all degrade performance. In practice, building a good RAG system requires careful engineering tradeoffs: choosing chunk sizes, balancing recall and precision, evaluating retrieval quality, and controlling cost and latency. Many failures that appear to be model failures are actually retrieval failures.

For production use, teams often add evaluation datasets, reranking, metadata filtering, caching, and citation support. These additions improve reliability and user trust, but they also increase system complexity. As a result, the best RAG systems are not necessarily the ones with the biggest models, but the ones with the most thoughtful end-to-end design.
